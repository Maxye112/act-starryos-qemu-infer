# **一、当前阶段完成情况**

本阶段围绕 Proj57 任务三开展工作，目标是在 QEMU 模拟的 RISC-V StarryOS 环境中完成 ACT 模型的 CPU 推理验证。目前已经完成模型导出与量化方案筛选、C++ 推理程序实现、StarryOS/QEMU 部署运行、单帧推理验证、闭环数据集评测以及运行时资源统计。从结果看，任务三所需的端到端流程已经跑通，并形成了可复现实验数据。

整体实现没有直接依赖训练环境，而是将训练后的 ACT 模型转为 ONNX，并在用户态通过 ONNX Runtime 进行 CPU 推理。这样做的好处是部署链路相对清晰，便于在 StarryOS/QEMU 环境中验证，也便于后续替换模型或继续做性能优化。

# **二、实现流程概述**

本阶段采用的流程可以概括为：先将 ACT 模型整理为适合部署的 ONNX 格式，再对模型做轻量化处理；随后编写 C++ 用户态推理程序，完成图像缩放、归一化、状态输入构造、ONNX Runtime 推理和动作反归一化；最后将程序、模型和必要参数放入 StarryOS 运行环境，在 QEMU 中完成验证。

推理输出是左右轮速度等动作信息。本报告中涉及“轮速差”的指标，是指左轮速度与右轮速度之差。该差值的符号可用于判断左转、右转或直行，因此既能评价数值误差，也能评价方向决策是否正确。

表 2 实现流程概述

|:--:|:--:|:--:|
| **阶段** | **做法** | **产出** |
| 模型准备 | 将训练模型导出为 ONNX，并比较不同量化方案 | 约 50 MB 的部署模型 |
| 推理程序 | 用 C++ 接入 ONNX Runtime，完成预处理、推理和动作解析 | RISC-V 用户态可执行程序 |
| 系统部署 | 在 StarryOS/QEMU 中运行推理程序 | 单帧运行结果和资源数据 |
| 结果验证 | 使用 666 帧数据进行闭环评测 | 误差、方向一致性和稳定性结果 |

# **四、模型方案对比**

本阶段比较了全精度模型、动作输出头保持半精度的模型，以及若干量化方案。“静态量化卷积/矩阵乘，动作头 FP16”表示主要计算层做 INT8 静态量化，而最终输出动作的部分保持 FP16，尽量兼顾体积和决策稳定性。

表 5 代表性模型方案对比

|  |  |  |  |
|:--:|:--:|:--:|:--:|
| **方案** | **轮速差平均绝对误差** | **误差≤0.01比例** | **转向符号准确率** |
| FP32 参考模型 | 0.0221 | 89.34% | 81.82% |
| 静态量化卷积/矩阵乘，动作头 FP16（当前选用） | 0.0198 | 89.79% | 92.31% |
| 动态量化注意力/前馈层，动作头 FP16 | 0.0213 | 84.08% | 57.58% |

综合比较后，当前选用第二种方案。它在模型体积上明显小于 FP32，同时轮速差误差更低；在模型给出转向时，左右符号准确率也更高，适合作为本阶段 StarryOS/QEMU 验证的部署模型。

# **五、运行资源与稳定性**

资源统计在 StarryOS/QEMU 环境中完成。由于原系统对进程内存字段支持不完整，本阶段对内核做了少量补充，使其能够像 Linux 一样读出常驻内存和峰值内存。该补充仅用于观测资源占用，不改变 ACT 推理逻辑。

表 6 StarryOS/QEMU 单帧资源统计

|                        |          |              |
|:----------------------:|:--------:|:------------:|
|        **阶段**        | **耗时** | **常驻内存** |
|        加载参数        | 10.5 ms  |    2.7 MB    |
| 创建 ONNX Runtime 环境 |  784 ms  |   10.3 MB    |
|      创建推理会话      | 8015 ms  |   75.8 MB    |
|       图像预处理       | 32.2 ms  |   76.4 MB    |
|        平均推理        | 5660 ms  |   76.7 MB    |

单进程峰值内存约 76.8 MB。推理耗时较高主要与 QEMU TCG 模拟环境有关，不等同于真实硬件上的最终性能。目前 666 帧闭环评测全程复用同一个推理会话，运行过程中未出现崩溃或异常退出，说明当前链路具备继续优化和扩展的基础。

# **六、当前问题与后续计划**

当前工作的主要不足是：第一，QEMU 模拟环境下推理延迟较高；第二，模型在转向判断上偏保守，部分真实转向帧被预测为直行；第三，目前验证重点集中在任务三的 CPU 路线，尚未扩展到 NPU/TPU 平台。
