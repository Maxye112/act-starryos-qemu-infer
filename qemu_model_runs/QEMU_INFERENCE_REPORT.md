### 4.1 关键结论

- **精度**：全 FP16 ≈ FP32（轮速差 MAE 差 < 0.001）；INT8+FP16 略大但仍可接受；动作头 FP16 与 FP32 几乎相同。
- **内存**：INT8+FP16 峰值约 **87 MB**，约为 FP32/全 FP16（~225 MB）的 **39%**。
- **耗时**：INT8+FP16 最快（~~6.9–8.3 s/帧）；全 FP16 最慢（~~20.8 s/帧），比 FP32 慢约 **18%**。
- **部署建议**：在 RISC-V CPU + ORT 场景下，**INT8+FP16 QDQ 是最佳平衡点**；全 FP16 无速度/内存优势。

## 7. 全 FP16 运行时的 ORT 警告

全 FP16 模型建 Session 时出现：

```
Could not find a CPU kernel and hence can't constant fold CumSum node
'/vision_encoder/encoder_cam_feat_pos_embed/CumSum'
```

原因：**ORT CPU EP 未注册 FP16 版 CumSum kernel**（仅支持 float/double/int）。常量折叠失败不影响推理完成，但会增加图优化阶段的警告。

## 8. ORT CPU 后端 FP16 支持分析（简要）


| 类别    | 算子示例                                     | CPU FP16 原生支持                     |
| ----- | ---------------------------------------- | --------------------------------- |
| 核心计算  | MatMul, Gemm, Softmax, Conv              | **否**（或依赖 x86 FP16 向量指令）          |
| 部分计算  | LayerNormalization, Add, Mul, Erf        | 部分有 MLFloat16 kernel，RISC-V 上常不可用 |
| 缺失    | CumSum, Sin, Cos                         | **否**                             |
| 布局/搬运 | Cast, Gather, Transpose, Reshape, Concat | **是**                             |


ORT 对无 FP16 kernel 的节点通过 `**InsertCastTransformer`** 自动插入 `fp16→fp32→fp16` 转换，用 FP32 kernel 兜底。这解释了：

1. 全 FP16 模型文件变小（97 MB），但运行时内存几乎不变（~228 MB）
2. 全 FP16 比 FP32 更慢（额外 Cast 开销 + QEMU 软件模拟）
3. INT8 QDQ 在 CPU 上有成熟 kernel，因此更快更省内存
